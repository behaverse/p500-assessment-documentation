# HANDOFF

Onboarding for anyone (human or agent) picking up this repo. Read this once, top to bottom, before making changes. The user-facing quick start lives in [README.md](README.md); this document is the deeper map.

## What this is

A documentation platform for the cognitive-assessment "engines" used in the **P500** study. The deliverable is a **static webapp** in [webapp/](webapp/) that renders content generated from two source-of-truth inputs (an Excel master spec and per-engine timeline configs). There is no server and no runtime build — the browser fetches generated data files and renders them with plain JS.

"Engine" = one cognitive task (e.g. BCS = Belval Card Sorting). There are **16** in the nav: BCS, DS, NB, WO, UFOV, TH, SRM, SOS, SMC, RE, BM, BSAC, MOT, OC, OOO, PC.

## Where to start

- **Run it:** `make serve`, then open http://localhost:8000 and click through a few engines, including a timeline page.
- **Change engine descriptions / parameter tables:** edit `content/Task spec.xlsx`, then `python3 scripts/build/excel_extractor.py` (regenerates `content/engines.json`).
- **Change timeline structure:** edit `content/timeline_configs/<ID>.json`, then `python3 scripts/generation/generate_timelines.py` (regenerates `webapp/pages/timelines/<engine>/*.html`).
- **Both at once:** `make build`.
- **Always verify in the browser** after changing generated content or JS — there is no test suite to catch regressions.

## Architecture

### Two independent pipelines feed one webapp

Everything is rooted in [content/](content/). The webapp is a thin client over two **separately generated** data products:

1. **Engine descriptions + parameters**
   `content/Task spec.xlsx` --[`scripts/build/excel_extractor.py`]--> `content/engines.json`
   `engines.json` is the **runtime data source** the webapp fetches at load. It carries each engine's `description` and `parameters` category content.

2. **Timeline pages**
   `content/timeline_configs/*.json` (+ `content/en.yaml` + `content/timeline_names.xlsx`)
   --[`scripts/generation/generate_enhanced_timeline_pages.py`]--> `webapp/pages/timelines/<engine>/<timeline>.html`
   Output is **pre-rendered HTML files** loaded on demand, not data merged into `engines.json`.

The two pipelines do not share output paths. Regenerate the one whose input you touched.

### Webapp runtime (no framework)

- [webapp/index.html](webapp/index.html) hardcodes the 16-engine nav list.
- [webapp/js/script.js](webapp/js/script.js) `fetch`es `content/engines.json` and drives the 3-tier nav: **engine -> category -> sub-item**. The HOME page content is hardcoded inside `script.js`, not in `engines.json`.
- Timeline content is loaded by `script.js` from `pages/timelines/<engine>/<timeline>.html` (lowercased path). [webapp/js/timeline.js](webapp/js/timeline.js) fetches `content/timeline_configs/<ENGINE>.json` and `content/en.yaml` to build the **list** of timelines per engine.

## Gotchas (read before editing)

- **The `webapp/content` symlink is load-bearing.** It points to `../content` so `fetch('content/engines.json')` resolves from the dev server rooted at `webapp/`. Don't replace it with a copy or delete it during cleanup. If it goes missing: `ln -s ../content webapp/content`.
- **Timeline configs are non-standard JSON.** Files in `content/timeline_configs/` contain C-style comments and trailing commas — they are **not valid JSON**. The generator strips them via regex in `EnhancedTimelineResolver._clean_json_comments` before parsing. Any new tool that reads these files must replicate that cleaning step (or reuse the resolver class).
- **Engine ID casing is inconsistent by directory.** IDs are uppercase in `engines.json`, the nav data attributes, and config filenames (`BCS.json`); but **lowercase** in timeline output paths (`webapp/pages/timelines/bcs/`). Parameter pages keep uppercase (`webapp/pages/parameters/BCS_parameters.html`). Match the existing casing per directory rather than guessing.
- **`engines.json` is a committed build artifact.** Treat it as generated, but it is tracked in git.
- **Run scripts from the repo root.** All Python paths are cwd-relative; `generate_timelines.py` is a wrapper that `chdir`s for you, the others are not.

## Known issues / open work

These are real, verified issues worth addressing — listed roughly by impact.

1. **`engines.json` bloat — RESOLVED (was ~9 MB).**
   The `timelines` category embedded fully-rendered timeline HTML (`subItems`) plus a duplicate `defaultContent` — ~8.5 MB the webapp never read (timeline content comes from `pages/timelines/*.html`; the timeline list comes from `timeline_configs/`; search reads only `categories[<cat>].content.body`). Fixed by `scripts/build/slim_engines.py`, which strips `subItems` and `defaultContent` and keeps `content`. Result: **9.03 MB -> 0.49 MB (95% smaller)**, with byte-identical search corpus and pixel-identical rendering. Re-run the script if `content/engines.json` is ever regenerated by the (currently divergent) build — see #2.

2. **The build does not reproduce `content/engines.json` (root cause of #1).**
   `excel_extractor.py` writes to `webapp/engines.json` (a path the app does not use; the file does not exist) and emits only `description` + `parameters` — no `timelines`. The runtime `content/engines.json` was produced by an older, now-absent process and is effectively a hand-maintained artifact. So `make build` does not regenerate the file the app loads. Proper fix (deferred): point `excel_extractor.py` at `content/engines.json`, merge in the timeline landing `content` from the timeline step, and emit the slim structure directly (no `subItems`) so the build is reproducible. Until then, `slim_engines.py` is the safety net.

3. **22 timeline configs vs 16 nav engines.** `content/timeline_configs/` has extras not in the nav: ML, RSAC, SART, SRT, SS, TOVA. Pages get generated for some of them (e.g. `pages/timelines/sart/`) but they are unreachable in the UI. Decide per config: promote into the 16-engine list, or remove to stop generating dead pages.

4. **No validation in CI.** `.github/workflows/deploy.yml` runs `make dist` and publishes with no checks. Given the non-standard JSON configs, a malformed config or a missing generated page ships silently. Cheap guards: `node -c webapp/js/*.js`, `python3 -c "import json; json.load(open('dist/content/engines.json'))"`, and an assertion that every timeline listed in a config has a generated HTML page.

5. **Nav duplication.** `webapp/index.html` hardcodes the engine list, duplicating `engines.json` keys — drift risk when adding/removing engines. Could be rendered from the fetched data.

6. **Stale `webapp/README.md`.** It describes categories `about`/`data` and "E1-E16"; the real categories are `description`/`parameters`/`timelines`. Update or delete it.

7. **Housekeeping.** `scripts/debug/` holds one-off debug scripts not part of the build; `update_webapp.py` is a stale legacy script (writes to `webapp/script.js`, references `migrated_content/`); `requirements.txt` uses unpinned `>=`. Archived material lives in `git_ignored/` (untracked).

8. **Placeholder content.** Every timeline page's "About" section is lorem ipsum, and MOT has an empty engine description (OC/PC/SMC are stubs). Full inventory in [CONTENT_TODO.md](CONTENT_TODO.md).

## Commands reference

```bash
pip install -r requirements.txt                      # build deps (first time)
make serve                                            # dev server at :8000
make build                                            # regenerate engines.json + timeline pages
python3 scripts/build/excel_extractor.py              # engines.json only (see issue #2: wrong output path)
python3 scripts/generation/generate_timelines.py      # timeline pages only
python3 scripts/build/slim_engines.py                 # strip unused timeline HTML from engines.json
make dist                                             # build deployable ./dist bundle
make clean                                            # remove ./dist
node -c webapp/js/script.js webapp/js/timeline.js     # JS syntax check (only quick check available)
```

There is no test suite, linter, or formatter wired up.

## Deployment plan

### Decision (current)

- **Repo scope:** ship this as a **P500-specific** repo now (e.g. `behaverse/p500-assessment-docs`). A separate general "Behaverse assessment documentation" site comes later, when that content exists. No premature abstraction in this repo.
- **Hosting:** **GitHub Pages**, served from the `behaverse` org. The Pages workflow already exists.

### Why P500-specific now

This content documents the exact assessment set used in one study. A general site would need a content model that spans studies/datasets, a way to label which assessments belong to which study, and likely a different information architecture. None of that is needed to publish the P500 docs. Building it now would be speculative. Keeping this repo single-purpose means the move to a general site later is a clean extraction, not a refactor under load. When the general site happens, this repo becomes either (a) one labeled collection imported into it, or (b) an archived study-specific site that the general site links to.

### Steps to deploy

1. **Create the repo** under the org: `behaverse/p500-assessment-docs` (or the org's preferred naming). Push `main`.
2. **Confirm the Pages workflow.** [.github/workflows/deploy.yml](.github/workflows/deploy.yml) already builds with `make dist` and deploys via `actions/deploy-pages`. In repo Settings -> Pages, set Source = "GitHub Actions".
3. **First deploy.** Push to `main` (or run the workflow manually via `workflow_dispatch`). The site publishes at `https://behaverse.github.io/p500-assessment-docs/`.
4. **Check relative paths.** The app fetches `content/...` and `pages/...` with **relative** URLs, so a project-path base (`/p500-assessment-docs/`) should work without changes. Verify after the first deploy that engine content, parameter pages, and at least one timeline page load — this is the most likely thing to break on Pages because of the symlink-to-copy swap and base path.
5. **(Recommended) Shrink the payload first.** Apply known issue #1 (strip embedded timeline HTML from `engines.json`) before or right after the first deploy so visitors aren't downloading ~9 MB. This is the single biggest user-facing improvement.
6. **(Optional) Add CI validation.** Known issue #3 — guards in the workflow so a broken build can't publish.

### Optional later

- **Custom domain** (e.g. `docs.behaverse.org` or a P500-specific subdomain) via a `CNAME` file + DNS, if the org wants it off `github.io`. Defer until the URL is decided org-wide.
- **General site extraction** when other studies' content is ready: lift the webapp shell and generators here into a content model keyed by study/dataset; import P500 as the first collection.
