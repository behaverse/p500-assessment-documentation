Glossary

Stimulus One ensemble of elements (their number is depending on StreamCount) that appear/play periodically and need to be matched to the n-previous stimuli.
Stream Sequence of stimuli of the same modality/feature. For example, in the standard dual N-back there’s a visual and an auditory stream.
Feature A feature of the stimulus that users have to pay attention to in order to determine if that feature-value is the same as the one N-trials back. Features include among others GridPosition, Sound, Color, Shape, Number and SpokenLetters. Each feature has a certain number of possible values (e.g., 8 positions on a grid). Across features that number might not be the same. It is common to use 8 feature levels.
Trial A trial comprises the presentation of a stimulus and possibly a user input and a feedback signal. A trial ends with the presentation of the next stimulus in the sequence.
Sequence A sequences defines a set of stimuli to be presented in a specific order. The Generative model focuses mainly on generating sequences that satisfy certain properties. Other names for “sequence” found in the literature are trial, block, run.
StreamCount The number of different features the user has to track (e.g. 2 in classic dual N-back: position on a grid & auditory letters).
TrialType Researchers typically distinguish different types of trials in the N-back task:
MatchTrial (aka “target” trial) trial where the current stimulus matches the stimulus presented N trials ago.
NonMatchTrial (aka “nontarget”) trial where the current stimulus is different from the one presented N trials ago.
LureTrial is a kind of NonMatchTrial; the trial would have a been a N-i or N+i match.
i = 1 recent lures
i > i non-recent lures
MatchTrialCount number of MatchTrial in a sequence of trials;
LureTrialCount number of LureTrial in a sequence of trials.

Parameters

Session parameters (1 parameters)

NValue

The “N” value, which determines how many trials back the user should compare the current stimulus to.

Type: integer Default: 1 Range: 1 - 7
Note: The max value of NValue is determined by the visual setup we chose; beyond that max we won’t be able to display all the boxes on the screen.

Stimulus parameters (6 parameters)

Streams

The definition of every stream the user has to track (1 for N-Back, 2 for Dual-N-Back, etc.).

Type: array of size [1,4] (defual = 1) of Feature

Feature

Which feature will be used

Type: enum Default: Letters

StimulusValue

The value of the current stimulus (as an index in the set depending on the feature type). Letters: 1 = A Digits: 1 = 1 Colors: The index of the color in a set of defined colors (depending on FeatureValuesCount) GridPosition8: The index of the lit cell (starting from top-left to bottom-right) GridPosition4: The index of the lit cell (starting from top-left to bottom-right)

Type: integer Default: 1 Range: 1 - 100

StreamSize

The number of stimuli inside a stream.

Type: integer Default: 30 Range: 1 - 200
Note: the maximum value is approximative. The current generation algorithm freezes the game for a noticeable duration for StreamSize > 200. Needs to be tested on low-end devices.

FeatureValuesCount

The number of different values to use for each feature. The same FeatureValuesCount applies to all Streams.

Type: integer Default: 8 Range: 1 - 100
Note: There are some limitations according to the Feature type for this variable. - Letters: FeatureValueCount <= 26 - Colors: FeatureValueCount <= 5 - GridPosition8: FeatureValueCount <= 8 - GridPosition4: FeatureValueCount <= 4 - Symbols: FeatureValueCount <= 5

RevealBoxContent

When we want to show the content of the boxes to the user.

Type: enum Default: StimulusOnly

User input parameters (3 parameters)

UseNonMatchButton

Whether we want to have a “Non-Match” button along with the “Match” button. The idea here is to disambiguate user input: when there are no non-match buttons, an absence of a response could mean the user reports a non-match or she did not pay attention to the stimulus.

Type: boolean Default: "false"
Note: If StreamCount > 1, the number of buttons are multiplied, therefore having a Non-Match button is discouraged for more than one stream.

PlayerPaced

Whether a user response is required to trigger the next trial.

Type: boolean Default: "false"
Note: Ignored if UseNonMatchButton = false

MaxResponseTime

The maximum time the user has to respond when PlayerPaced is true. After that, the trial is considered failed.

Type: float Default: 5 Range: 0 - Inf
Note: Ignored if PlayerPaced = false If StimulusDuration > MaxResponseTime, MaxResponseTime is ignored and user can answer for the whole StimulusDuration

Feedback parameters (5 parameters)

ShowAccuracyOnButtonFeedback

Whether we color the outline of the buttons red or green to indicate whether the response was correct or incorrect. If false, outline will be white.

Type: boolean Default: "true"

ShowMissedTrialFeedback

Whether a feedback is shown in the case when there is no NonMatchButton, the trial is a matchTrial and the user didn’t press the MatchButton during the period separating the current stimulus presentation and the onset of the next stimulus presentation (i.e., elapsed time > MaxResponseTime).

Type: boolean Default: "false"
Note: If set to True, this Feedback presentation should be followed by InterStimulusInterval and not the immediate presentation of the next stimulus which would be confusing. If ShowMissedTrialFeedback = True, show central red cross.

ShowAccuracyCheckFeedback

Whether we should show the valid or wrong “check” feedback at the end of a trial.

Type: enum Default: Never

ShowCorrectTrialResponseFeedback

Whether we should show the correct response feedback at the end of a trial.

Type: enum Default: OnError
Note: To visualize this variable, after finishing the answer period, In case of UseNonMatchButton= false, if the user was supposed to press the equal button but failed to do so (i.e., miss), then the button border will light up with yellow color.

RevealNBackBoxContentFeedback

Whether we should show the content of the N-Back box at the end of a trial so the player can compare with the current stimulus.

Type: enum Default: OnError

Temporal parameters (4 parameters)

StimulusDisplayDuration

How long each stimulus should be displayed. When stimuli are auditory files (e.g., the word “cat”), the sounds should be played only once in integrity (and not be repeated to fill up StimulusDuration). Any stimulus sound should be cut before playing another.

Type: float Default: 0.5 Range: 0.01 - Inf

InterStimulusInterval

Time interval starting after the presentation of a stimulus and ending with either - presentation of the feedback (then foreperiod, then the next stimulus) - presentation of the foreperiod (then the next stimulus) When the task is PlayerPaced, the user response will trigger the presentation of the feedback and thus shortcut the InterStimulusInterval.

Type: float Default: 2.5 Range: 0 - Inf
Note: InterStimulusInterval is sometimes NOT the actual delay between the offset of a stimulus and the onset of the next. It turns out that in the general case this cannot be easily set given the way things are currently implemented (e.g., unknown how long different feedback types last, additional delays added by the state machines). It should be possible to compute the InterStimulusInterval using the events; we might then need to manually adjust the value of InterStimulusInterval so we get the actual ISI we want.

TrialForeperiod

A time interval between feedbacks and the onset of the next trial. When feedbacks are absent, the TrialForeperiod still exists.

Type: float Default: 0.5 Range: 0 - Inf

StimulusMovingAnimationDuration

The duration of the sliding animation within the stimuli queue.

Type: float Default: 0.1 Range: 0 - Inf
Note: If it is bigger than StimulusDisplay then the amount will set to StimulusDisplay itself.
