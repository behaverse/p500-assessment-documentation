# Behaverse Assessment Documentation

## Executive Summary

This specification defines a comprehensive web-based documentation platform for the behaverse cognitive assessments. It is important to keep in mind the following terminology:
 - engine: is a piece of software that can be parameterized to create a specific cognitive assessment. One engine typically suppoorts multiple different cognitive assessments.
 - timeline: is a specific configuration of an engine. It sets values for the engine parameters and defines instructions for the users of that test. 

The platform will transform scattered documents (including Excel-based data) into an organized, inter-linked, searchable, and user-friendly web interface.





## Problem Statement

### Current State
- Assessment documentation exists across multiple disconnected documents
- Researchers struggle to find relevant parameter specifications
- No centralized demo access or timeline configuration examples
- Data dictionaries are buried in complex spreadsheets
- Scientific justifications and references are scattered across multiple documents

### Pain Points
- **Discovery**: Users cannot easily find the right assessment for their needs
- **Configuration**: Parameter specifications are hard to navigate and understand
- **Implementation**: No clear path from documentation to actual assessment deployment
- **Validation**: Sample datasets and expected outputs are not readily accessible
- **Maintenance**: Updates to documentation require manual Excel file management

### User Needs
- **Researchers**: Need to quickly identify suitable assessments and understand their parameters
- **Clinicians**: Require clear instructions and validated configurations for specific use cases
- **Developers**: Need comprehensive API documentation and sample data for integration
- **Content Managers**: Require streamlined workflows for updating documentation


## Solution Architecture

### Technical Stack Decision
**Primary Recommendation: Quarto + GitHub Pages**
- **Rationale**: Combines documentation generation with interactive elements
- **Benefits**: Version control, automated builds, extensible with custom components
- **Alternative**: Consider Docusaurus or GitBook for more complex interactivity needs

### System Architecture
```
Data Sources → Content Pipeline → Web Platform → User Interface
     ↓              ↓              ↓              ↓
- Task spec.xlsx   - Automated    - Static site  - Responsive web
- Timeline configs   parsing       - CDN hosted   - Search enabled
- Media assets     - Validation   - SSL secured  - Mobile optimized
```

### Core Platform Components

#### 1. Homepage
- **Purpose**: Project overview and navigation entry point
- **Content**: 
  - Brief description of Behaverse assessment documentation
  - Quick start guide for different user types
  - Featured/popular assessments
- **Requirements**: Fast loading, clear navigation, responsive design

#### 2. Engine Directory
- **Purpose**: Comprehensive listing of all assessment engines
- **Layout**: Grid view with engine cards showing:
  - Engine thumbnail (GIF or static image)
  - Engine name and category
  - Brief description (50-100 characters) accessible when clicking on the image 
  - Quick stats (# parameters, # timelines, last updated)
- **Features**: 
  - Filtering by category, complexity, or use case
  - Sorting by popularity, alphabetical, or last updated
  - Search functionality across engine names and descriptions

#### 3. Individual Engine Pages
Each engine requires a dedicated page with the following sections:

**Header Section:**
- Engine name and category
- Hero image/animation demonstrating the engine
- Brief description (2-3 sentences)
- Demo link (opens in new tab/embedded iframe)

**Parameters Section:**
- **Data Source**: Parsed from Task spec.xlsx
- **Display Format**: 
  - Searchable/filterable table with columns: Parameter Name, Type, Default Value, Range, Description
  - Advanced view: Include dependencies, validation rules, impact on results
  - Export functionality (CSV, JSON)
- **User Experience**: 
  - Collapsible sections for parameter categories
  - Tooltip explanations for technical terms
  - Visual indicators for required vs optional parameters

**Timelines Section:**
- **Purpose**: Pre-configured parameter sets for specific use cases
- **Content Structure**:
  - Timeline name and scientific justification
  - Parameter configuration summary
  - Expected duration and participant requirements
  - Links to relevant research papers
  - Download links for config.json files
- **Display**: Card-based layout with preview functionality

**Data Dictionary Section:**
- **Content**: All variables generated by the engine
- **Format**: Interactive table with:
  - Variable name, type, description, example values
  - Statistical properties (range, distribution, etc.)
  - Correlation with other variables
- **Features**:
  - Sample dataset download
  - Data visualization previews
  - Export documentation in multiple formats

#### 4. Timeline Detail Pages
**Requirements for dedicated timeline pages:**
- **Justification**: Detailed scientific rationale
- **Configuration**: Complete parameter specifications
- **Instructions**: User-facing text formatted for clarity
- **Demo Integration**: Embedded or linked demonstration
- **Usage Examples**: Real-world application scenarios
- **Download Section**: Config files, instruction sets, sample data 


## Technical Requirements

### Extract the data from the relevant sources and reorganize them in a meaningful way so that it can be easily digested by the webapp.

### Performance Requirements
- **Page Load Time**: < 3 seconds for any page
- **Search Response**: < 500ms for parameter/data dictionary searches

### System Requirements
- **Hosting**: Static site deployment (GitHub Pages, Netlify, or Vercel)
- **CDN**: Global content distribution for media assets
- **SSL**: HTTPS encryption required

### Accessibility Requirements
- **WCAG 2.1 AA Compliance**: Full accessibility support
- **Screen Readers**: Compatible with major screen reading software
- **Keyboard Navigation**: Full functionality without mouse
- **Color Contrast**: Minimum 4.5:1 ratio for all text
- **Mobile Responsive**: Functional on devices from 320px width
- **Styling**: use a minimalistic but stylish font and color palette. 

### Security Considerations
- **Content Security Policy**: Strict CSP headers
- **Data Privacy**: No personal data collection without consent
- **Input Validation**: Sanitize all search inputs
- **External Links**: Security warnings for demo links

## Data Management Strategy

### Content Sources
1. **Primary Data**: `content/Task spec.xlsx`
   - Engine parameters and specifications
   - Data dictionary definitions
   - Validation rules and constraints

2. **Configuration Files**: Timeline JSON configurations
   - Parameter value sets
   - Scientific justifications
   - Usage instructions

3. **Media Assets**: `visual_references/` directory
   - Engine demonstration images/GIFs
   - Instructional diagrams
   - Timeline-specific visuals

### Content Pipeline
1. **Automated Parsing**: Script to convert Excel data to structured format (YAML/JSON)
2. **Validation Layer**: Ensure data integrity and completeness
3. **Version Control**: Git-based workflow for all content updates
4. **Build Process**: Automated site generation on content changes
5. **Quality Assurance**: Link checking and content validation

### Update Workflow
1. **Content Manager** updates source files in repository
2. **Automated validation** checks data integrity
3. **Preview deployment** for review and testing
4. **Production deployment** after approval
5. **User notification** of significant changes
